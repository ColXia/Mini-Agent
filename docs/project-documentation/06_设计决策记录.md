# Mini-Agent 设计决策记录

## 1. 概述

本文档记录 Mini-Agent 项目的关键设计决策，包括决策背景、考虑的选项、最终决策及理由。

---

## 2. 架构决策

### 2.1 [ADR-001] 采用六边形架构

**状态**: 已采纳

**背景**

Mini-Agent 需要支持多种用户界面（CLI、Desktop、Gateway）和多种 LLM 提供商（Anthropic、OpenAI、OpenRouter 等）。需要一种架构模式来解耦核心业务逻辑与外部依赖。

**考虑的选项**

1. **分层架构** - 传统三层架构
2. **六边形架构** - Ports and Adapters 模式
3. **洋葱架构** - 同心圆分层
4. **Clean Architecture** - 依赖规则分层

**决策**

采用**六边形架构**。

**理由**

- 清晰的端口定义，便于依赖倒置
- 适配器模式天然支持多 Surface 和多 Provider
- 核心业务逻辑完全独立
- 易于测试，可 Mock 所有外部依赖

**影响**

- 所有 Port 定义为 Protocol
- Application 层依赖 Port，Infrastructure 层实现 Port
- Surface 层作为入站适配器，LLM Client 作为出站适配器

---

### 2.2 [ADR-002] 使用 dataclass 定义数据合约

**状态**: 已采纳

**背景**

需要选择一种数据结构定义方式，用于定义 Agent、Run、Session 等核心数据合约。

**考虑的选项**

1. **Pydantic BaseModel** - 数据验证 + 序列化
2. **dataclass** - 轻量级数据类
3. **attrs** - 增强的数据类
4. **NamedTuple** - 命名元组

**决策**

使用 **`@dataclass(frozen=True, slots=True)`**。

**理由**

- `frozen=True` 确保不可变性，符合函数式设计
- `slots=True` 减少内存占用，提升性能
- 原生 Python，无额外依赖
- Pydantic 用于 API 边界验证，内部使用 dataclass

**影响**

- 所有合约类不可变，状态变更通过创建新实例
- 需要手动处理序列化/反序列化
- 使用 `field(default_factory=...)` 处理可变默认值

---

### 2.3 [ADR-003] 异步优先设计

**状态**: 已采纳

**背景**

Agent 执行涉及大量 I/O 操作（LLM API 调用、文件操作、网络请求），需要选择并发模型。

**考虑的选项**

1. **同步阻塞** - 简单但性能差
2. **多线程** - GIL 限制，线程安全复杂
3. **asyncio 异步** - 单线程并发，适合 I/O 密集
4. **多进程** - CPU 密集场景

**决策**

采用 **asyncio 异步优先**设计。

**理由**

- Agent 执行主要是 I/O 密集（LLM API、文件、网络）
- 单线程并发避免 GIL 限制
- 流式输出天然支持
- 与 FastAPI、Anthropic SDK 等生态兼容

**影响**

- 所有 I/O 操作使用 async/await
- 使用 `asyncio.Event` 实现取消机制
- 需要处理异步上下文和异常

## 3. 模型管理决策

### 3.1 [ADR-004] Agent-Model 服务边界分离

**状态**: 已采纳

**背景**

Agent 需要使用 LLM 服务，但不应关心模型选择、提供商配置、故障转移等细节。

**决策**

**Agent 侧只消费模型服务接口，Model Manager 侧处理所有复杂性**。

**理由**

- 关注点分离：Agent 专注于任务执行
- 配置集中：模型配置在 Model Manager 统一管理
- 故障隔离：模型故障不影响 Agent 核心逻辑
- 易于扩展：新增 Provider 不影响 Agent 代码

**影响**

- Agent 通过 `AgentModelPort` 获取模型服务
- Model Manager 负责路由选择、熔断、故障转移
- 模型能力通过 `ModelCapabilities` 暴露给 Agent

---

### 3.2 [ADR-005] 三层模型路由

**状态**: 已采纳

**背景**

需要支持多提供商、多模型，并支持故障转移和负载均衡。

**决策**

采用**三层模型路由**：Provider Catalog → Route Selector → Circuit Breaker。

**理由**

- 分层设计，职责清晰
- Provider Catalog 管理配置
- Route Selector 处理选择逻辑
- Circuit Breaker 提供熔断保护

**影响**

- 配置文件 `providers.json` 定义提供商
- 运行时根据优先级和健康状态选择路由
- 熔断器自动隔离故障 Provider

---

## 4. 工具系统决策

### 4.1 [ADR-006] 权限分级模型

**状态**: 已采纳

**背景**

工具执行需要权限控制，不同工具有不同的风险级别。

**决策**

采用**四级权限模型**：SAFE、NORMAL、DANGEROUS、RESTRICTED。

**理由**

- 分级清晰，便于管理
- SAFE 级别自动批准，减少用户干扰
- DANGEROUS 级别必须审批，确保安全
- RESTRICTED 级别需要特殊权限

**影响**

- 工具定义时指定权限级别
- 权限引擎根据级别和策略决策
- 用户可配置审批策略

---

### 4.2 [ADR-007] 沙箱执行环境

**状态**: 已采纳

**背景**

工具执行可能涉及文件操作、Shell 命令等危险操作，需要隔离执行环境。

**决策**

采用**沙箱执行**：FileSandbox + ShellSandbox。

**理由**

- 文件操作限制在工作空间边界内
- Shell 命令白名单控制
- 防止意外或恶意操作
- 记录所有变更，支持回滚

**影响**

- 所有文件操作通过 FileSandbox
- Shell 命令通过 ShellSandbox
- MutationLedger 记录所有变更

---

## 5. 技能系统决策

### 5.1 [ADR-008] 三层技能解析

**状态**: 已采纳

**背景**

技能可能来自多个来源（内置、全局、工作空间），需要解决命名冲突。

**决策**

采用**三层技能解析**：Internal (core.*) → Global (global.*) → Workspace (ws.*)。

**理由**

- 前缀区分来源，避免冲突
- 优先级：Workspace > Global > Internal
- 工作空间技能可覆盖全局技能
- 内置技能提供基础能力

**影响**

- 技能名称使用前缀：`core.file_operations`、`global.custom_skill`
- 无前缀时按优先级解析
- SkillResolver 处理冲突

---

### 5.2 [ADR-009] SKILL.md 技能定义格式

**状态**: 已采纳

**背景**

需要一种易于编写和维护的技能定义格式。

**决策**

采用 **SKILL.md Markdown 格式**定义技能。

**理由**

- Markdown 易于编写和阅读
- YAML Front Matter 定义元数据
- Markdown 正文定义指令
- 版本控制友好

**影响**

- 技能目录结构：`skills/{name}/SKILL.md`
- 加载器解析 Front Matter 和正文
- 支持技能继承和组合

---

## 6. 会话管理决策

### 6.1 [ADR-010] JSONL 会话存储

**状态**: 已采纳

**背景**

会话数据需要持久化，支持增量写入和快速恢复。

**决策**

采用 **JSONL (JSON Lines)** 格式存储会话。

**理由**

- 增量写入，无需重写整个文件
- 每行一个事件，易于解析
- 支持流式读取大文件
- 便于日志分析和调试

**影响**

- 会话文件：`.mini-agent/sessions/{session_id}.jsonl`
- 每个事件一行 JSON
- 支持压缩和归档

---

### 6.2 [ADR-011] 三层记忆系统

**状态**: 已采纳

**背景**

Agent 需要记忆能力，但上下文窗口有限。

**决策**

采用**三层记忆系统**：Working Memory → Short-term Memory → Long-term Memory。

**理由**

- Working Memory：当前对话，在上下文窗口内
- Short-term Memory：近期会话，快速访问
- Long-term Memory：持久化，向量检索

**影响**

- Context Compaction 将 Working Memory 压缩
- Memory Consolidation 将 STM 迁移到 LTM
- LTM 使用向量数据库存储

---

## 7. 运行时决策

### 7.1 [ADR-012] 事件驱动架构

**状态**: 已采纳

**背景**

运行时需要处理各种事件（Run 状态变更、工具调用、审批请求等），需要解耦事件生产者和消费者。

**决策**

采用**事件驱动架构**：EventBus + Event Handlers。

**理由**

- 解耦组件，降低复杂度
- 支持多个消费者订阅同一事件
- 易于扩展新的事件处理器
- 支持异步事件处理

**影响**

- 所有状态变更发布事件
- Surface 层订阅事件更新 UI
- 支持事件持久化和回放

---

### 7.2 [ADR-013] 检查点恢复机制

**状态**: 已采纳

**背景**

长时间运行的 Agent 可能需要暂停、恢复或从错误中恢复。

**决策**

采用**检查点机制**：定期保存执行状态，支持恢复。

**理由**

- 支持暂停/恢复
- 支持错误恢复
- 支持回滚到历史状态
- 便于调试和分析

**影响**

- 每个步骤后保存检查点
- 检查点链式存储
- 恢复时加载最新检查点

---

## 8. 设计原则总结

| 原则 | 说明 |
|------|------|
| 依赖倒置 | 上层依赖 Port，下层实现 Port |
| 单一职责 | 每个模块专注于单一领域 |
| 开闭原则 | 通过 Port 扩展，不修改现有代码 |
| 接口隔离 | Port 定义最小必要接口 |
| 不可变性 | 数据合约使用 frozen dataclass |
| 异步优先 | I/O 操作使用 asyncio |
| 事件驱动 | 状态变更发布事件 |
| 配置分离 | 配置与代码分离 |
