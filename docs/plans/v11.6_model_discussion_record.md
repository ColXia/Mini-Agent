## 一、讨论背景

用户希望了解模型管理模块的设计，并提出两个增强需求：
1. 模型池支持详细的参数配置

## 二、需求澄清过程

### 2.1 模型参数配置

**用户需求**: 模型池支持详细的参数配置

**澄清问题**: 参数配置的层级？

| 选项 | 说明 | 用户选择 |
|------|------|---------|
| A | 按 Provider 全局配置 | ❌ |
| B | 按 Model 单独配置 | ✅ |
| C | 按 Agent 覆盖配置 | ❌ |

**进一步澄清**: 一个模型是否需要多个参数配置？

**用户回答**: 是的，同一个模型可以有多个参数配置。比如两个侧重不一样的 Agent 使用同一个模型，但拥有各自优化调整的模型参数配置。

**设计结论**:
- 参数配置独立于模型定义
- 每个配置有唯一 ID（如 "sonnet-code"、"sonnet-creative"）
- Agent 绑定时绑定的是"模型+配置"的组合

---

### 2.2 路由选择能力匹配

**用户需求**: 路由支持多模态、推理、结构化输出、流式输出

**澄清问题**: 路由时如何处理能力需求？

| 选项 | 说明 | 用户选择 |
|------|------|---------|
| A | 自动匹配路由（不支持就换模型） | ❌ |
| B | 模型能做什么 = Agent 能做什么，不支持就不支持 | ✅ |
| C | 降级处理（不支持就用基础能力） | ❌ |

**设计结论**:
- 不做复杂的能力匹配路由
- 模型能力探测结果只做"信息展示"
- 告诉用户"这个模型支持什么能力"
- 不影响路由逻辑

## 四、Provider 配置管理讨论 (第二轮)

### 4.1 模型接入方式

**问题**: 模型配置/接入是怎么实现的？

**选项**:
- A) 按协议分为两套 (OpenAI/Anthropic)
- B) 按云和本地分
- C) URL + API Key + 协议类型

**用户选择**: C - 走 url + api_key + 协议类型，没必要搞复杂

**结论**: 
- 只区分 openai/anthropic 两种协议类型
- 不按云/本地分，也不按服务商分
- 配置格式统一：name, api_base, api_key, api_type

---

### 4.2 配置管理简化

**用户需求**: 简化设计，以用户主动添加为主

**讨论要点**:
1. 预设供应商支持 - 用户认为未来迭代再说
2. 环境变量检测 - 只检测官方标准变量名
3. 模型列表 - 用户手动配置或主动触发探测

**用户确认**:
- 环境变量检测：只检测 OPENAI_API_KEY, ANTHROPIC_API_KEY, MINIMAX_API_KEY 等官方变量名
- 用户可选择使用环境变量，也可选择完全手动配置
- 模型列表：不配置就提示配置，不自动探测

**最终设计**:
```
Provider 配置:
├── 必填: name, api_base, api_key, api_type
├── 可选: models, timeout, priority, headers, enabled
└── models 为空时提示用户配置

环境变量检测:
├── 只检测官方变量名
├── 用户可选择使用或不使用
└── 手动配置优先级更高

模型列表:
├── 用户手动配置
├── 用户主动触发探测
└── 不配置则提示，使用时返回"无模型"
```

## 五、最终设计方案

### 5.1 Provider 配置结构

**必填字段 (4个)**:
| 字段 | 说明 |
|------|------|
| name | 显示名称 |
| api_base | API 地址 |
| api_key | API 密钥 |
| api_type | 协议类型 (openai/anthropic) |

**可选字段**:
| 字段 | 默认值 |
|------|--------|
| models | [] (需配置) |
| timeout | 60 |
| priority | 0 |
| headers | {} |
| enabled | true |

### 5.2 环境变量检测

| 环境变量 | 服务商 |
|---------|--------|
| OPENAI_API_KEY | OpenAI |
| ANTHROPIC_API_KEY | Anthropic |
| MINIMAX_API_KEY | MiniMax |

**优先级**: 手动配置 > 环境变量检测

### 5.3 模型列表管理

| 方式 | 说明 |
|------|------|
| 手动配置 | 用户填写 models 字段 |
| 触发探测 | `mini-agent provider detect-models <name>` |
| 不配置 | 提示"请先配置可用模型" |

### 5.4 CLI 命令

```bash
mini-agent provider add/list/show/validate/update/delete
mini-agent provider detect-models <name>
mini-agent config create/update/delete/list/show
mini-agent detect-capabilities <model_id>
```

## 六、Agent 与模型职责边界讨论 (第三轮)

### 6.1 Agent 需要知道的信息

**用户确认**: Agent 需要知道以下信息才能正常工作

| 信息类型 | 用途 |
|---------|------|
| 模型能力 | Agent 判断模型是否支持某功能 |
| 当前状态 | Agent 判断模型是否可用、是否需要转移 |
| 错误详情 | Agent 判断如何处理错误 (重试/转移/报错) |
| Token 消耗 | Agent 判断是否需要压缩上下文 |

### 6.2 故障转移职责

**用户确认**: 故障转移是 Agent 侧的功能

- 用户配置 Agent 时选择多个模型
- Agent 发现模型不可用时自动转移
- 收到熔断通知后触发转移
- 熔断器是模型侧的，负责监控和通知

### 6.3 模型绑定逻辑

**用户确认**: 一个 Agent 可以绑定多个模型

- 配置保存在 Agent 侧
- 用户从模型池选取模型配置到 Agent
- 支持主模型和备用模型

### 6.4 模型发现

**用户确认**: 用户触发式

- 不自动发现
- 用户点击触发按钮后才获取

### 6.5 Token 成本估算

**用户确认**: 模型侧负责，但用户可自定义定价

- 用户可查看 Provider 下具体模型消耗
- 模型成本由用户自行设定
- 考虑中转站等特殊定价场景

## 八、最终设计方案汇总

### 8.1 模型侧设计

| 模块 | 内容 |
|------|------|
| Provider 配置 | 必填 4 字段，可选 models 等 |
| 环境变量检测 | 只检测官方变量名，可选使用 |
| 参数配置多配置 | 同一模型可有多个参数配置 |
| 自定义名称 | 覆盖原名显示 |
| 模型能力探测 | 手动触发，结果展示 |
| Token 统计 | 记录消耗，用户自定义定价 |
| 健康监控 | 监控成功率、延迟 |
| 熔断器 | 连续失败后熔断，通知 Agent |

### 8.2 Agent 侧设计

| 模块 | 内容 |
|------|------|
| 模型绑定 | 一个 Agent 绑定多模型 |
| 故障转移 | 收到熔断通知后自动切换 |
| 错误处理 | 根据错误类型决定处理方式 |
| 上下文压缩 | 根据 Token 消耗触发压缩 |

### 8.3 ModelServicePort 接口

```python
class ModelServicePort(Protocol):
    # 绑定与客户端
    get_binding(agent_id) → ModelBindingView
    get_client(binding) → LLMClient
    switch_model(agent_id, target) → ModelBindingView
    list_available_models() → list[ModelOptionView]
    
    # 能力查询
    get_capabilities(binding) → ModelCapabilitiesView
    
    # 状态查询
    get_health_status(model_config_id) → HealthStatusView
    get_breaker_status(model_config_id) → BreakerStatusView
    
    # 错误信息
    get_last_error(model_config_id) → ErrorDetail
    
    # Token 消耗
    get_token_usage(model_config_id) → TokenUsageView
    
    # 回调注册
    on_breaker_triggered(callback)
    on_health_changed(callback)
```
